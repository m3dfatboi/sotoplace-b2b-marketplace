# Contributing to Sotoplace

## Процесс разработки

### 1. Создание новой фичи

1. Создать ветку от `main`:
```bash
git checkout -b feature/название-фичи
```

2. Внести изменения

3. Написать тесты

4. Запустить линтеры:
```bash
make format
make lint
```

5. Запустить тесты:
```bash
make test
```

6. Создать коммит:
```bash
git commit -m "feat: описание фичи"
```

7. Отправить в репозиторий:
```bash
git push origin feature/название-фичи
```

8. Создать Pull Request

### 2. Исправление бага

1. Создать ветку от `main`:
```bash
git checkout -b fix/название-бага
```

2. Исправить баг

3. Добавить тест, воспроизводящий баг

4. Убедиться, что тест проходит

5. Создать коммит:
```bash
git commit -m "fix: описание исправления"
```

## Стиль коммитов

Используем [Conventional Commits](https://www.conventionalcommits.org/):

- `feat:` - новая функциональность
- `fix:` - исправление бага
- `docs:` - изменения в документации
- `style:` - форматирование кода
- `refactor:` - рефакторинг
- `test:` - добавление тестов
- `chore:` - обновление зависимостей, конфигурации

Примеры:
```
feat: add user authentication
fix: resolve order calculation bug
docs: update API documentation
refactor: simplify order service logic
test: add tests for product creation
```

## Стандарты кода

### Python
- Следуем PEP 8
- Используем type hints везде
- Максимальная длина строки: 100 символов
- Используем black для форматирования
- Используем ruff для линтинга

### Документация
- Все публичные функции должны иметь docstrings
- API endpoints должны иметь описания
- Сложная логика должна быть прокомментирована

### Тесты
- Покрытие тестами должно быть > 80%
- Тесты должны быть изолированными
- Используем pytest fixtures
- Тесты должны быть быстрыми

## Code Review

### Что проверяем
- [ ] Код соответствует стандартам
- [ ] Есть тесты
- [ ] Тесты проходят
- [ ] Документация обновлена
- [ ] Нет breaking changes (или они задокументированы)
- [ ] Производительность не ухудшилась
- [ ] Безопасность не нарушена

### Процесс review
1. Автоматические проверки (CI/CD)
2. Code review от минимум 1 разработчика
3. Тестирование на staging
4. Merge в main
5. Deploy на production

## Вопросы?
